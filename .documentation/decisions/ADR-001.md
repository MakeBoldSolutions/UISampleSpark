# ADR-001: Data-Core Project Consolidation

## Status

Accepted

## Context

The solution originally maintained a separate `UISampleSpark.Data` project housing EF Core entities,
database context, and repository implementations. As the educational codebase matured, the separation
added build complexity and cross-project reference overhead without providing meaningful isolation
benefits for a single-domain demo application.

## Decision

Merged `UISampleSpark.Data` into `UISampleSpark.Core`, consolidating all domain models, interfaces,
repositories, services, and EF Core data access into a single Core project. All dependent projects
(`UISampleSpark.UI`, `UISampleSpark.MinimalApi`, `UISampleSpark.CLI`, test projects) were updated
to reference only `UISampleSpark.Core`.

## Consequences

### Positive

- Reduced solution from 5 projects to 4, lowering build time and cognitive overhead
- Eliminated cross-layer circular reference risk between Data and Core
- Simplified DI registration — one assembly to scan for repository/service implementations
- Aligns with the educational goal of demonstrating clean architecture without unnecessary ceremony

### Negative

- Stricter separation of data access from domain logic is no longer enforced by project boundaries
- Teams that fork the repo for production use will need to re-introduce the Data layer split manually
